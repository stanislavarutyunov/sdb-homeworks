### Задание 1. Резервное копирование

### Кейс
Финансовая компания решила увеличить надёжность работы баз данных и их резервного копирования. 

Необходимо описать, какие варианты резервного копирования подходят в случаях: 

1.1. Необходимо восстанавливать данные в полном объёме за предыдущий день.

В данном случае подойдет либо полное,либо дифференциальное резервное копирование. Я бы использовал второй вариант,потому что система дифференциального резервного копирования экономит много места для хранения и ускоряет процесс резервного копирования по сравнению с полным резервным копированием. 
Например, когда вы создаете дифференциальную резервную копию в первый день, создается полная резервная копия. На второй день будет создана резервная копия всех изменений, внесенных с первого дня. А на 1-й день вы также создадите резервную копию всех изменений, сделанных с первого дня. Как вы можете догадаться, на 3-й день будет дифференциальная резервная копия, содержащая изменения, сделанные во 3-й и XNUMX-й день. Дифференциальная резервная копия очень полезна при добавлении данных постоянно.

1.2. Необходимо восстанавливать данные за час до предполагаемой поломки.

Здесь бы я использовал инкрементное резервное копирование.
Система инкрементного резервного копирования создана для смягчения недостатков системы полного резервного копирования. Когда вы создаете инкрементную резервную копию, все, что вы делаете, - это резервное копирование всех изменений с момента последней резервной копии. Например, когда вы создаете инкрементную резервную копию в первый день, программа резервного копирования создаст начальную полную резервную копию. На второй день программа создаст резервную копию изменений, внесенных с первого дня. На третий день инкрементная резервная копия данных будет содержать только изменения, внесенные со второго дня и так далее.

### Задание 2. PostgreSQL

2.1. С помощью официальной документации приведите пример команды резервирования данных и восстановления БД (pgdump/pgrestore).

# PGDUMP:

# Выгрузка базы данных mydb в файл SQL-скрипта:
```
$ pg_dump mydb > db.sql

Например:

$ pg_dump zabbix.db > /tmp/zabbix.db.sql
```
# Выгрузка базы данных в специальном формате:
```
$ pg_dump -Fc mydb > db.dump

Например:

$ pg_dump -Fc zabbix.db > /tmp/zabbix.db.dump
```
# PGRESTORE

Предположим, что мы выгрузили базу данных mydb в файл специального формата:
```
$ pg_dump -Fc mydb > db.dump

Например:

$ pg_dump -Fc zabbix.db > /tmp/zabbix.db.dump
```
Мы можем удалить базу данных и восстановить её из копии:

```
$ dropdb mydb
$ pg_restore -C -d postgres db.dump

Например:

$ dropdb zabbix.db
$ pg_restore -C -d postgres zabbix.db.dump

```
2.1.* Возможно ли автоматизировать этот процесс? Если да, то как?

*Приведите ответ в свободной форме.*


Нижеприведенный пример скрипта выполняет бэкап и удаляет все файлы старше 61 дня (за исключением 15-го числа месяца).

```
#!/bin/sh

PATH=/etc:/bin:/sbin:/usr/bin:/usr/sbin:/usr/local/bin:/usr/local/sbin

PGPASSWORD=some_password
export PGPASSWORD
pathB=/mnt/backup
dbUser=dbadmin
database=zabbix


find $pathB \( -name "*-1[^5].*" -o -name "*-[023]?.*" \) -ctime +61 -delete
pg_dump -U $dbUser $database | gzip > $pathB/pgsql_$(date "+%Y-%m-%d").sql.gz


unset PGPASSWORD
```
Чтобы настроить регулярное выполнение, выполним следующую команду в планировщике crontab:
```
# crontab -e
3 0 * * * /etc/scripts/pgsql_dump.sh # postgres pg dump
```

  # Percona XtraBackup:
  
 После создания полного бэкапа, на его основе можно сделать инкрементный, где будут только изменения со времени полного бэкапа
 
 ```
 # xtrabackup --backup --target-dir=/root/backupdb/inc1 --incremental-basedir=/root/backupdb/full
 ```
 Можно проверить, что конкретно в себя включает полный бэкап, а что инкремент. В директории с бэкапами есть файл xtrabackup_checkpoints примерно следующего содержания.

```
backup_type = full-backuped
from_lsn = 0
to_lsn = 17687056
last_lsn = 17687065
compact = 0
recover_binlog_info = 1
flushed_lsn = 17687065
```
LSN - log sequence number. Это регистрационные номера транзакций. В данном случае полный бэкап начинается с нулевой транзакции и заканчивается 17687056. Теперь смотрим этот же файл в директории inc1.
```
backup_type = incremental
from_lsn = 17687056
to_lsn = 17710039
last_lsn = 17710048
compact = 0
recover_binlog_info = 1
flushed_lsn = 17710048
```
Инкрементный бэкап базы начался с той транзакции, на которой закончился полный. Следующий инкрементный архив может быть создан опять на базе полного, либо уже инкрементного. То есть либо так:
```
# xtrabackup --backup --target-dir=/root/backupdb/inc2 --incremental-basedir=/root/backupdb/full

либо так:

# xtrabackup --backup --target-dir=/root/backupdb/inc2 --incremental-basedir=/root/backupdb/inc1
```
